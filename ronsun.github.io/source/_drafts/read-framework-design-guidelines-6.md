### 擴充性機制
不夠的要加隨時都能輕鬆加，但多加的要刪時一定都是破壞性變更。  

> 不只擴充性，在軟體開發上都是這樣，不只是對外開放的功能或 API，其他部分 (例如資料庫欄位) 要加時也都要謹慎考慮刪除時的容易度，如果加了很難刪，那就應該考慮清楚確認必要且正確後再加。

#### 非密封類別 (Unsealed Classes) 的設計
+ **O 考慮** 使用無 `virtual` / `protected` 成員的非密封類別提供擴充性。  
    VANCE MORRISON 表示這件事的重點是謹慎考慮，只要 unsealed 後就難以回頭且會無法套用某些最佳化機制，而設計上對外承諾可擴充的意思。  

#### `protected` 成員
+ **O 應該** 在安全性、文件化、相容性分析的角度將其視為公開成員層級。
    > 對繼承的子類別來說，是可以完全存取 `protected` 成員的，所以要假設會被依賴 (安全性)、需要文件以幫助使用者 (文件化)、難以在無破壞性變更下修改 (相容性)。從這些場景來看幾乎就是公開成員的層級了。

#### 事件 (Event) 與回呼 (Callback)
回呼指的是委派。

+ **O 應該** 優先使用事件而不是一般回呼，因為大多數開發者對事件模型更熟悉，工具支援也更完整。

+ **X 避免** 在效能敏感的情境下使用回呼。

+ **O 應該** 在設計回呼 API 時優先使用 `Func<...>`、`Action<...>` 或 `Expression<...>`，而不是自訂委派。

RICO MARIANI 表示一般先用 `Func<...>` / `Action<...>` 就好，除非是需要分析、序列化、或在其他程序中評估才會使用 `Expression<...>`。  
> 就是除非需要用到 Expression Tree (例如 LINQ to SQL 用來翻譯 SQL) 否則預設用 `Func<...>` / `Action<...>` 就好。

+ **O 應該** 知道當呼叫委派方法時，其實是在執行任意程式碼，這會影響安全性、正確性、以及相容性。
> 安全性：執行由呼叫端傳入的委派方法，本質上是執行任何呼叫端決定的程式碼。  
> 正確性：呼叫端傳入的方法拋例外，或是改變參考型別的參數內容。
> 相容性：呼叫端方法在框架內重複觸發或更改觸發條件順序時，可能產生預期外的結果，可能是破壞性變更。

BRIAN PEPIN 提供了一段 Windows Forms team 的範例，說明了在 lock 區塊中呼叫包含一個有 lock 的委派方法，可能會造成死鎖，而委派方法由呼叫端決定，框架開發者無法預期。

#### `virtual` 成員
這類成員和回呼在擴充性上很接近，但執行效能與記憶體使用更佳。

+ **X 避免** 在無強烈理由下設計 `virtual` 成員，這是基於它的設計、測試與維護成本。

+ **O 應該** 優先讓 `protected virtual` 成為擴充點，而不是直接公開 `public virtual` 成員。

+ **O 應該** 讓公開成員負責固定對外契約，再委派給 `protected virtual` 成員提供客製化空間。

#### 抽象 (Abstractions) - 抽象型別與介面
+ **X 避免** 在經過多個實際衍生類別的實作且驗證前提供抽象化。
> 抽象化，例如介面，沒開好的話就容易變成沒用或難用的東西，要刪或要改的成本都非常高，對大型熱門框架來說甚至是不可能。

+ **O 考慮** 提供一套測試案例來讓使用者驗證是否正確實作該抽象型別或介面。


### 基底類別 (Base Classes) 的設計
+ **O 考慮** 不管有沒有抽象成員都把基底類別設計成抽象類別，讓使用者明確知道這是設計來被繼承的。

+ **O 考慮** 將基底類別放在與主要使用情境型別不同的命名空間，因為基底類別設計上是為了提供擴充性，一般的使用者通常不會關注。

+ **X 避免** 在公開 API 中暴露只對抽象實作者有價值的基底類別；如果某個基底類別只是為了重用實作細節，通常不該成為主要 API 的一部分。

+ **X 避免** 在公開 API 中以 `Base` 為後綴命名基底類別。原因是設計上更偏好用基底類別做為回傳型別 (即使該類別是設計來被繼承的)，但使用者並不需要在意這個是不是基底類別，多了 `Base` 後綴會增加使用者的認知負擔。

> 說是這樣說，但 .NET 自己就一堆基底類別是用 Base 後綴，我還一直以為這是慣例。  
> 從另一個角度來說，如果真的很難從語意上區分基底和衍生類別，那在找不到適合的命名時，我還是會選擇後綴 Base 來命名基底類別。

### 密封 (Sealing) 的設計
+ **X 避免** 密封類別，除非有很好的理由，很好的理由包含
    - 靜態類別
    - 類別在可被繼承存取的 `protected` 成員中保存了安全敏感資訊
    - 不希望這個類別被繼承，而它又繼承了一堆 `virtual` 成員，那就應該直接密封整個類別，而不是逐個成員封鎖。
    - 對於需高效查詢的特性 (Attribute)，應優先設為密封 (`sealed`)，以降低執行期型別判斷成本。

+ **O 考慮** 對已覆寫的成員再加上 `sealed`，把可擴充範圍限制在真正需要的層級。

+ **X 不應** 在密封型別中宣告 `protected` 或 `virtual` 成員，因為這些擴充點已經不可能被繼承使用。

> 和非密封相反，兩者都有各自的優缺點，唯一相同的就是了解優缺點後，適當的決定是否密封。

### 結論
這章的核心其實是 "要用多貴的方式提供多少擴充性"。因為擴充點一旦公開，就等於承諾了長期維護責任，之後要收回幾乎都會變成破壞性變更。
